## üî≠ Project Overview

CUA implements a **perceive ‚Üí think ‚Üí act** loop: it captures a screenshot of a virtual Linux desktop, sends the image to a vision-language model alongside the user's task description, receives a structured action command, and executes that command on the desktop. This cycle repeats until the task is complete, an error is unrecoverable, or the step limit is reached.

### Key Capabilities

| | Capability | Implementation |
|:---:|---|---|
| üåê | Browser automation (accessibility-tree) | Playwright MCP server via JSON-RPC |
| ‚ôø | Semantic desktop automation | AT-SPI2 accessibility framework via GObject Introspection |
| ü§ñ | Native computer-use protocol | Gemini CU + Claude CU structured tool calls (model-driven action loop) |
| üìπ | Real-time screen streaming | WebSocket screenshots, WebRTC via ffmpeg x11grab + aiortc, noVNC |
| üßÝ | Multi-provider AI (model allowlist) | UI is restricted to 4 models (served by `GET /api/models`): `claude-sonnet-4-6`, `claude-opus-4-6`, `gemini-3-flash-preview`, `gemini-3.1-pro-preview` |

### Problem It Solves

CUA enables natural-language-driven computer control ‚Äî a user describes a task in plain text (e.g., "Open Firefox and search for weather in New York"), and the agent autonomously operates the desktop to complete it, reporting progress in real time through a web UI.

## ü§ñ Model Policy (UI + API)

The UI dropdown is intentionally restricted to **four** models:

- **Anthropic**: `claude-sonnet-4-6`
- **Anthropic**: `claude-opus-4-6`
- **Google**: `gemini-3-flash-preview`
- **Google**: `gemini-3.1-pro-preview`

**Single source of truth:** `backend/allowed_models.json`  
**Frontend source:** `GET /api/models`  
**Backend enforcement:** `POST /api/agent/start` rejects any provider/model not in the allowlist.

### Add more models later (extensible by design)
1. Edit `backend/allowed_models.json`
2. Restart the backend
3. The UI auto-refreshes supported models from `GET /api/models`

> Tip: keep the allowlist small by default. It makes demos reliable and avoids "model drift" across machines.

## üîÑ System Flow

### End-to-End Execution: User Input ‚Üí Final Output

```mermaid
flowchart TD
    A["User enters task + selects engine/provider in UI"] --> B["Frontend POST /api/agent/start"]
    B --> C{"Backend validates inputs\n(rate limit + concurrency, engine, provider, model allowlist, API key)\nErrors: 400 (validation) / 429 (rate or concurrency)"}
    C -->|invalid| C1["Return error to UI"]
    C -->|valid| D["Start Docker container if not running"]
    D --> E["Create AgentLoop with callbacks"]
    E --> F["Spawn asyncio task ‚Üí run loop"]

    F --> G["PERCEIVE: Capture screenshot"]
    G --> H["THINK: Send screenshot + task + history to LLM"]
    H --> I["Parse JSON action from model response"]
    I --> J{"Action type?"}

    J -->|"done"| K["Mark session COMPLETED"]
    J -->|"error"| L["Mark session ERROR"]
    J -->|normal action| M["ACT: Validate ‚Üí Normalize ‚Üí Dispatch to engine"]

    M --> N["Engine executes action inside container"]
    N --> O{"3 consecutive errors?"}
    O -->|yes| L
    O -->|no| P{"Agent stuck\n(N identical actions)?"}
    P -->|yes| Q["Inject recovery hint + WAIT"]
    P -->|no| R{"Max steps reached?"}
    R -->|yes| K
    R -->|no| G

    K --> S["Broadcast agent_finished via WebSocket"]
    L --> S
    S --> T["UI shows final status"]
```

### Step-by-Step Breakdown

1. **User submits task** via the Workbench UI (`Workbench.jsx`), selecting provider (Google/Anthropic), model, engine, and entering the task description.
2. **Frontend calls** `POST /api/agent/start` with `StartTaskRequest` payload (`api.js`).
3. **Backend validates** rate limit (10 starts/min), engine membership in `SUPPORTED_ENGINES`, provider/model allowlists, API key resolution (UI ‚Üí .env ‚Üí system env), max concurrent sessions (‚â§3), and task non-emptiness (`server.py`).
4. **Docker container starts** automatically via `docker_manager.start_container()` if not already running.
5. **`AgentLoop`** is instantiated with callbacks for log, step, and screenshot events that broadcast to all WebSocket clients (`server.py`).
6. **Pre-flight checks** run per engine: agent service health for all, MCP initialization for `playwright_mcp`, AT-SPI bus check for `omni_accessibility` (`loop.py`).
7. **Perceive**: `capture_screenshot()` calls the agent service (`GET :9222/screenshot?mode={mode}`) or falls back to `docker exec scrot` (`screenshot.py`).
8. **Think**: `query_model()` routes to `query_gemini()` or `query_claude()` with the screenshot, task, step number, last 15 actions as history, and the engine-specific system prompt (`model_router.py`).
9. **Parse**: The raw model text is stripped of markdown fences, JSON is extracted (handling nested braces), action aliases are resolved, and an `AgentAction` is validated (`gemini_client.py` / `anthropic_client.py`).
10. **Act**: `execute_action()` validates the engine, normalizes the action via `UnifiedAction`, checks engine capability schema, validates coordinates (‚â§1440√ó900) and text (‚â§5000 chars), then dispatches to the correct engine (`executor.py`).
11. **Termination**: The loop ends on `DONE` action, `ERROR` action, 3 consecutive step errors, user stop request, cancellation, or reaching `max_steps`.
12. **Real-time updates**: Each step, log entry, and screenshot is broadcast to WebSocket clients. The UI renders a live timeline, log viewer, and screen view.

## üìä Agent Loop Logic

### Core Loop (`backend/agent/loop.py`)

`AgentLoop.run()` executes up to `max_steps` iterations. Each step runs through `_execute_step()` with a per-step timeout (`step_timeout`, default 30s).

### Failure Recovery

- **Consecutive error tracking**: A counter resets on each successful step and increments on errors. At 3 consecutive errors (`MAX_CONSECUTIVE_ERRORS`), the session aborts with `ERROR` status.
- **Duplicate action detection**: `_detect_stuck()` checks whether the last N actions have the same action type, similar coordinates, and same text. If stuck, a `WAIT` action with a recovery hint is injected. Parameters are engine-aware:
  - **Browser** (`playwright_mcp`): lookback = 3 actions, coordinate tolerance = 10 px (`MAX_DUPLICATE_ACTIONS`, `loop.py:L704`)
  - **Desktop** (`omni_accessibility` / `computer_use`): lookback = 2 actions, coordinate tolerance = 30 px (`MAX_DUPLICATE_ACTIONS_DESKTOP`, `DESKTOP_COORD_TOLERANCE`, `loop.py:L43-44`)
  - **Maximum stuck detections**: After 3 stuck-detection firings (`MAX_STUCK_DETECTIONS`), the loop force-terminates to prevent burning steps on ignored recovery hints.
  - **Duplicate result detection**: If the last 2+ execution results are identical (e.g., same `evaluate_js` output), an ultimatum to return `done`/`error` is injected (`MAX_DUPLICATE_RESULTS = 2`).
- **Recovery hints** (`_build_recovery_hint()`): Engine-specific remediation advice is injected as the reasoning for the WAIT action. For example, a stuck `CLICK` triggers advice to use `evaluate_js`, `key Enter`, or `scroll_to`. A stuck `FILL` suggests discovering field names via JS evaluation.

### State Machine

```mermaid
stateDiagram-v2
    [*] --> Idle
    Idle --> Running : POST /api/agent/start
    Running --> StepPerceive : next step
    StepPerceive --> StepThink : screenshot captured
    StepThink --> StepAct : action parsed
    StepAct --> StepPerceive : action executed
    StepAct --> Completed : DONE action
    StepAct --> Error : ERROR action
    StepAct --> Error : 3 consecutive errors
    Running --> Completed : max_steps reached
    Running --> Completed : user stop request
    Running --> Completed : asyncio cancellation
    Completed --> [*]
    Error --> [*]
```

## ‚öôÔ∏è Engine Breakdown

Three engines are defined in `SUPPORTED_ENGINES` (sourced from `engine_capabilities.json`):

### 1. `playwright_mcp` ‚Äî Browser Automation (Accessibility-Tree-Based)

- **Dispatch**: `executor.py` ‚Üí `playwright_mcp_client.execute_mcp_action()` ‚Üí JSON-RPC over HTTP to the Playwright MCP server on port 8931
- **Runtime**: Standalone MCP server (`@playwright/mcp`) managing its own browser instance
- **Actions**: click, fill, type, navigate, tabs (by accessible name/role, not coordinates)
- **System prompt**: `SYSTEM_PROMPT_PLAYWRIGHT_MCP` ‚Äî targets elements by name/role/label
- **Special**: Uses `find_element` to get accessibility snapshots; coordinates are ignored

### 2. `omni_accessibility` ‚Äî Cross-Platform Semantic Desktop Automation

- **Dispatch**: `executor.py` ‚Üí `accessibility_engine.execute_accessibility_action()` ‚Üí AT-SPI2 (Linux) / UI Automation (Windows) / JXA (macOS) + xdotool for physical input
- **Runtime**: Platform auto-detected ‚Äî GObject Introspection (Linux), PowerShell UIA (Windows), osascript JXA (macOS)
- **Actions**: find elements by role/name/state, click by element ID, type into focused elements, get accessibility tree, navigate tree nodes, window activation
- **Features**: TTL element cache, circuit breaker, semantic scoring, post-action verification, cross-platform provider abstraction
- **Requirements**: Linux: `at-spi2-core`, `python3-gi`, D-Bus | Windows: PowerShell 5.1+ | macOS: Accessibility permissions

### 3. `computer_use` ‚Äî Native Computer-Use Protocol (Gemini CU / Claude CU)

- **Dispatch**: `loop.py` ‚Üí `_run_computer_use_engine()` ‚Üí `ComputerUseEngine.execute_task()` (bypasses `executor.py` entirely ‚Äî the model controls the action loop)
- **Runtime**: Uses provider-native computer-use tools.
  - **Gemini**: `computer_use` tool with screenshot/action loop (normalized coordinates).
  - **Claude**: tool version is **auto-detected** (e.g., `computer_20250124` vs `computer_20251124` depending on model), using `client.beta.messages.create(..., betas=[...])`. `ClaudeCUClient` also accepts explicit `tool_version`/`beta_flag` params from `allowed_models.json` metadata.
- **Execution modes**: Browser (via Playwright CDP page) or Desktop (via xdotool + scrot)
- **Coordinate handling**: Gemini sends normalized 0‚Äì999 coordinates (denormalized to pixels by the executor); Claude sends real pixel coordinates
- **Actions**: `click_at`, `double_click_at`, `type_text_at`, `key_combination`, `scroll_at`, `move_mouse`, `drag_to`, `wait`, `screenshot`, `navigate_to`, `go_back`, `go_forward`, `select_option`, `hover_at`, `done`
- **Features**: Model-driven screenshot loop, safety decision handling (`require_confirmation`), inline base64 screenshot feedback, thinking/reasoning traces
- **Contract**: Uses its own internal turn loop ‚Äî never dispatched through `execute_action()`

### Engine Dispatch Diagram

```mermaid
flowchart LR
    EX["executor.py"]

    EX -->|"playwright_mcp"| MCP["playwright_mcp_client\nJSON-RPC :8931"]
    EX -->|"omni_accessibility"| A11Y["accessibility_engine\nAT-SPI/UIA/JXA"]

    LP["loop.py"] -->|"computer_use"| CUE["ComputerUseEngine\nNative CU protocol"]
```

### Engine Isolation

The executor enforces **strict engine isolation**: the user-chosen engine is used for every action in the session. No fallback to other engines occurs at the executor level. The `computer_use` engine bypasses the executor entirely ‚Äî it runs its own model-driven loop via `ComputerUseEngine`. This is verified by 15 tests in `tests/test_engine_isolation.py`.

## üîê Security Model

### Input Validation (`backend/api/server.py`, `backend/agent/executor.py`)

| Protection | Implementation |
|---|---|
| **Rate limiting** | Sliding-window limiter: max 10 `POST /api/agent/start` calls per 60 seconds (`_RateLimiter` class) |
| **Concurrent session cap** | Max 3 active sessions (`_MAX_CONCURRENT_SESSIONS`) |
| **Max steps hard cap** | `max_steps` capped at 200 regardless of user input (`_MAX_STEPS_HARD_CAP`) |
| **Engine allowlist** | Request engine validated against `SUPPORTED_ENGINES` set |
| **Provider allowlist** | Only `google` and `anthropic` accepted (`_VALID_PROVIDERS`) |
| **Model allowlist** | Per-provider model validation (`_VALID_MODELS_BY_PROVIDER`) |
| **Session ID validation** | UUID format validation before any session lookup (`_is_valid_uuid()`) |
| **Task length limit** | Max 10,000 characters (`StartTaskRequest.task` field) |
| **API key masking** | Keys are masked in audit logs (`sk-a...9f2e` format) |
| **Coordinate bounds** | Validated ‚â§1440√ó900, non-negative integers |
| **Text length limit** | Max 5,000 characters per action |

### Container Security (`docker/agent_service.py`)

| Protection | Implementation |
|---|---|
| **Request body size** | Max 1 MB per request (`_MAX_BODY_SIZE`) |
| **Blocked shell commands** | Pattern list: `rm -rf /`, `mkfs.`, `dd if=/dev/`, `shutdown`, `reboot`, etc. (`_BLOCKED_CMD_PATTERNS`) |
| **Command allowlist** | Only specific commands permitted in `run_command` action: `ls`, `cat`, `grep`, `python3`, `curl`, etc. (`_ALLOWED_COMMANDS` frozenset) |
| **File upload path restriction** | Only `/tmp`, `/app`, `/home` directories allowed (`_UPLOAD_ALLOWED_PREFIXES`) |

### Container Isolation (`docker-compose.yml`)

| Setting | Value |
|---|---|
| `security_opt` | `no-new-privileges:true` |
| `shm_size` | `2gb` |
| Memory limit | `4g` |
| CPU limit | `2` cores |
| Port binding | `127.0.0.1` only (not exposed externally) |

### CORS (`backend/api/server.py`)

Restricted to local development origins: `localhost:5173`, `127.0.0.1:5173`, `localhost:3000`, `127.0.0.1:3000`. Methods limited to `GET` and `POST`.

### What the Security Model Does NOT Cover

- No authentication or authorization on the API (any local client can start agents)
- No TLS on inter-service HTTP (backend ‚Üí agent service is plain HTTP over localhost)
- The `evaluate_js` action executes arbitrary JavaScript inside the browser with no sandboxing beyond the browser's own security context
- The `run_command` allowlist does not prevent all possible abuse (e.g., `python3 -c "..."` can run arbitrary code)
- WebSocket connections have no authentication

## üßÝ LLM Provider Integration

### Supported Providers & Models (UI Allowlist)

| Provider | Models in UI (allowlist) | Env Variable |
|---|---|---|
| Google Gemini | `gemini-3-flash-preview`, `gemini-3.1-pro-preview` | `GOOGLE_API_KEY` |
| Anthropic Claude | `claude-sonnet-4-6`, `claude-opus-4-6` | `ANTHROPIC_API_KEY` |

The UI options come from `GET /api/models` (loaded from `backend/allowed_models.json`).

### Model Selection

The user selects a provider and model in the UI. `model_router.query_model()` dispatches to the matching client. There is no automatic fallback between providers ‚Äî if the selected provider fails, the step records an error.

### Request Construction

Both clients follow the same pattern:
1. Build a context window with the last 15 actions as history (trimmed for token economy)
2. Attach the current screenshot as an inline image (base64 PNG)
3. Include the task description and current step number
4. Set the engine-specific system prompt from `prompts.py`

### Response Parsing

Both clients use the same parsing logic:
1. Strip markdown fences (` ```json ... ``` `)
2. Extract the first valid JSON object (handling nested braces)
3. Resolve action aliases via `resolve_action()`
4. Validate that the action exists in `ActionType`
5. Return `(AgentAction, raw_response_text)`

### Retry Logic

The Gemini client retries up to `gemini_retry_attempts` (default 3) with `gemini_retry_delay` (default 2s) between attempts on transient failures. The Anthropic client follows the same pattern.

### API Key Resolution (`backend/config.py`)

Keys are resolved in priority order:
1. **UI input** ‚Äî user pastes key in the Workbench
2. **`.env` file** ‚Äî `GOOGLE_API_KEY` or `ANTHROPIC_API_KEY` in project root `.env`
3. **System environment** ‚Äî same variable names as system env vars

The `/api/keys/status` endpoint reports which sources are available for each provider, including masked key previews.

## üê≥ Docker Runtime

### Container: `cua-environment`

Built from `docker/Dockerfile` on Ubuntu 24.04. The entrypoint (`docker/entrypoint.sh`) starts the following services in order:

1. **D-Bus** ‚Äî system + session bus (required for AT-SPI)
2. **Xvfb** ‚Äî virtual X11 framebuffer at `:99`, resolution `1440√ó900√ó24`
3. **AT-SPI** ‚Äî accessibility bridge + registry daemon (before desktop so apps register correctly)
4. **XFCE 4** ‚Äî full desktop environment with window manager
5. **x11vnc** ‚Äî VNC server on port 5900 (optional `VNC_PASSWORD` for authentication)
6. **noVNC + websockify** ‚Äî browser-accessible VNC on port 6080
7. **Browser bootstrap** ‚Äî pre-warms Chrome profile, sets default browser via `xdg-settings`
8. **ydotoold** ‚Äî ydotool daemon (if `/dev/uinput` is available)
9. **Playwright MCP server** ‚Äî `@playwright/mcp` HTTP transport on port 8931 (headless, `--no-sandbox`)
10. **Agent Service** ‚Äî `agent_service.py` HTTP server on port 9222

### Pre-installed Software

- Google Chrome (stable), Firefox
- Playwright Chromium + Firefox browsers
- Playwright MCP server (`@playwright/mcp` via npm)
- xdotool, wmctrl, xclip, ydotool, scrot
- AT-SPI2 accessibility stack (`at-spi2-core`, `gir1.2-atspi-2.0`, `python3-gi`)
- ffmpeg (for video capture)

## üîß Troubleshooting

### "Agent Service Down" / timeouts (Windows)
- Ensure Docker ports are reachable on IPv4:
  - Use `127.0.0.1` (not `localhost`) for `AGENT_SERVICE_HOST` and `PLAYWRIGHT_MCP_HOST`.
- Quick checks:
  - `curl http://127.0.0.1:9222/health`
  - Open `http://127.0.0.1:6080`

### Start rejects with 400/429
`POST /api/agent/start` returns:
- **400** for validation problems (invalid model/provider/engine/task/api key)
- **429** for rate limiting or concurrency cap
The frontend surfaces these as `{ error: "..." }` messages.

### noVNC loads but desktop is black
- Wait ~5‚Äì15 seconds after container start (XFCE session boot)
- Check container logs: `docker compose logs -f`

## ‚öÝÔ∏è Limitations

The following are constraints observed directly in the codebase:

1. **In-memory session state only** ‚Äî no persistence. Backend restart loses all active sessions (`server.py` uses plain dicts).
2. **Model allowlist (intentional)** ‚Äî the UI exposes only the models listed in `backend/allowed_models.json`. This keeps demos stable; edit the allowlist to add more.
3. **No authentication** ‚Äî the API accepts requests from any local client without auth tokens. WebSocket connections are unauthenticated.
4. **Coordinate-dependent engines** ‚Äî `computer_use` (in desktop mode) relies on the model correctly predicting pixel coordinates from screenshots. Accuracy depends on model vision quality.
5. **AT-SPI availability** ‚Äî the `omni_accessibility` engine requires the D-Bus session bus and AT-SPI2 bindings to be running inside the container (Linux). On Windows/macOS it uses native platform APIs. Applications that don't expose accessibility trees are invisible to this engine.
6. **Fixed viewport** ‚Äî the display resolution is fixed at 1440√ó900 (set in Dockerfile and config). The coordinate validation and system prompts assume this size.
7. **Sequential engine execution** ‚Äî each session uses exactly one engine. There is no mechanism to switch engines mid-session or combine engine capabilities.
8. **Rate limiting is in-memory** ‚Äî the sliding-window rate limiter resets on backend restart.
9. **WebRTC max 2 connections** ‚Äî `_MAX_CONNECTIONS = 2` in `webrtc_server.py`.
10. **Context window truncation** ‚Äî only the last 15 actions are sent to the model. Long sessions lose early context.
11. **`run_command` security** ‚Äî while the allowlist and blocked-pattern list exist, `python3 -c "..."` is permitted and can execute arbitrary code inside the container.

## üîÆ Future Improvements

Based on patterns observable in the codebase:

1. **Persistent session storage** ‚Äî the `AgentSession` Pydantic model is structured for serialization. Adding a database or file-backed store is a natural extension.
2. **Additional model support** ‚Äî the `model_router.py` dispatch pattern and `_VALID_MODELS_BY_PROVIDER` allowlist are designed for easy extension to new models.
3. **Wayland support** ‚Äî `engine_capabilities.json` already declares `wayland_support: true` for the `omni_accessibility` engine. The schema is future-proofed.
4. **`allowed_domains` enforcement** ‚Äî the `StartTaskRequest` model accepts an `allowed_domains` list, but no enforcement logic exists in the agent loop or executor yet.
5. **Custom system prompts** ‚Äî the `StartTaskRequest` accepts a `system_prompt` field, but it is not currently passed through to the model clients.
6. **Engine certification CI gate** ‚Äî `backend/health/engine_certifier.py` implements a full certification framework. Integrating it into CI would gate deployments on engine health.

Line 296: ## üìä Agent Loop Logic
Line 298: ### Core Loop (`backend/agent/loop.py`)
Line 300: `AgentLoop.run()` executes up to `max_steps` iterations. Each step runs through `_execute_step()` with a per-step timeout (`step_timeout`, default 30s).
Line 302: ### Failure Recovery
Line 304: - **Consecutive error tracking**: A counter resets on each successful step and increments on errors. At 3 consecutive errors (`MAX_CONSECUTIVE_ERRORS`), the session aborts with `ERROR` status.
Line 305: - **Duplicate action detection**: `_detect_stuck()` checks whether the last N actions have the same action type, similar coordinates, and same text. If stuck, a `WAIT` action with a recovery hint is injected. Parameters are engine-aware:
Line 306: - **Browser** (`playwright_mcp`): lookback = 3 actions, coordinate tolerance = 10 px (`MAX_DUPLICATE_ACTIONS`, `loop.py:L704`)
Line 307: - **Desktop** (`omni_accessibility` / `computer_use`): lookback = 2 actions, coordinate tolerance = 30 px (`MAX_DUPLICATE_ACTIONS_DESKTOP`, `DESKTOP_COORD_TOLERANCE`, `loop.py:L43-44`)
Line 308: - **Maximum stuck detections**: After 3 stuck-detection firings (`MAX_STUCK_DETECTIONS`), the loop force-terminates to prevent burning steps on ignored recovery hints.
Line 309: - **Duplicate result detection**: If the last 2+ execution results are identical (e.g., same `evaluate_js` output), an ultimatum to return `done`/`error` is injected (`MAX_DUPLICATE_RESULTS = 2`).
Line 310: - **Recovery hints** (`_build_recovery_hint()`): Engine-specific remediation advice is injected as the reasoning for the WAIT action. For example, a stuck `CLICK` triggers advice to use `evaluate_js`, `key Enter`, or `scroll_to`. A stuck `FILL` suggests discovering field names via JS evaluation.

Line 930: ## ‚öÝÔ∏è Limitations
Line 932: The following are constraints observed directly in the codebase:
Line 934: 1. **In-memory session state only** ‚Äî no persistence. Backend restart loses all active sessions (`server.py` uses plain dicts).
Line 935: 2. **Model allowlist (intentional)** ‚Äî the UI exposes only the models listed in `backend/allowed_models.json`. This keeps demos stable; edit the allowlist to add more.
Line 936: 3. **No authentication** ‚Äî the API accepts requests from any local client without auth tokens. WebSocket connections are unauthenticated.
Line 937: 4. **Coordinate-dependent engines** ‚Äî `computer_use` (in desktop mode) relies on the model correctly predicting pixel coordinates from screenshots. Accuracy depends on model vision quality.
Line 938: 5. **AT-SPI availability** ‚Äî the `omni_accessibility` engine requires the D-Bus session bus and AT-SPI2 bindings to be running inside the container (Linux). On Windows/macOS it uses native platform APIs. Applications that don't expose accessibility trees are invisible to this engine.
Line 939: 6. **Fixed viewport** ‚Äî the display resolution is fixed at 1440√ó900 (set in Dockerfile and config). The coordinate validation and system prompts assume this size.
Line 940: 7. **Sequential engine execution** ‚Äî each session uses exactly one engine. There is no mechanism to switch engines mid-session or combine engine capabilities.
Line 941: 8. **Rate limiting is in-memory** ‚Äî the sliding-window rate limiter resets on backend restart.
Line 942: 9. **WebRTC max 2 connections** ‚Äî `_MAX_CONNECTIONS = 2` in `webrtc_server.py`.
Line 943: 10. **Context window truncation** ‚Äî only the last 15 actions are sent to the model. Long sessions lose early context.
Line 944: 11. **`run_command` security** ‚Äî while the allowlist and blocked-pattern list exist, `python3 -c "..."` is permitted and can execute arbitrary code inside the container.

Line 948: ## üîÆ Future Improvements
Line 950: Based on patterns observable in the codebase:
Line 952: 1. **Persistent session storage** ‚Äî the `AgentSession` Pydantic model is structured for serialization. Adding a database or file-backed store is a natural extension.
Line 953: 2. **Additional model support** ‚Äî the `model_router.py` dispatch pattern and `_VALID_MODELS_BY_PROVIDER` allowlist are designed for easy extension to new models.
Line 954: 3. **Wayland support** ‚Äî `engine_capabilities.json` already declares `wayland_support: true` for the `omni_accessibility` engine. The schema is future-proofed.
Line 955: 4. **`allowed_domains` enforcement** ‚Äî the `StartTaskRequest` model accepts an `allowed_domains` list, but no enforcement logic exists in the agent loop or executor yet.
Line 956: 5. **Custom system prompts** ‚Äî the `StartTaskRequest` accepts a `system_prompt` field, but it is not currently passed through to the model clients.
Line 957: 6. **Engine certification CI gate** ‚Äî `backend/health/engine_certifier.py` implements a full certification framework. Integrating it into CI would gate deployments on engine health.
